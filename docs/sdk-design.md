# RouteIQ SDK Design

## Why the generated SDKs are not the developer interface

The `routeiq-sdk-{lang}` repos contain **protobuf bindings** — auto-generated from
`routeiq-schema` by `buf generate`. They are the correct source of truth for the
**wire format**: every language emits identical `routeiq.*` OTel attributes, the field
names never drift, and a schema change propagates to all seven languages in one CI run.

But generated proto bindings are not a developer interface. Using them directly
requires understanding:

- The `AgentEvent` envelope and how to nest `TaskEvent`, `StepEvent`, `ToolCallEvent`
- The `conventions/telemetry.yaml` attribute mapping
- OTel provider setup, `BatchSpanProcessor`, `OTLPSpanExporter`
- Proto enum values (`entities_pb2.TOOL_SUCCESS`, `events_pb2.TASK_STARTED`, …)

That is framework knowledge, not user knowledge. An instrumented agent written against
the raw generated SDK is ~100 lines of boilerplate per agent file.

## The two-layer model

```
┌─────────────────────────────────────────────────────┐
│  Developer layer  — hand-written, idiomatic per lang │
│  routeiq.RouteIQ / RouteIQ.Client / riq.New(...)     │
│  context managers · callbacks · fluent builders      │
└───────────────────────┬─────────────────────────────┘
                        │ uses
┌───────────────────────▼─────────────────────────────┐
│  Wire layer  — generated by buf from routeiq-schema  │
│  routeiq-sdk-{python,go,ts,java,ruby,rust,csharp}   │
│  proto types · OTel attribute keys · enum values     │
└─────────────────────────────────────────────────────┘
```

The **wire layer** is auto-generated and never hand-edited.  
The **developer layer** is hand-written inside each SDK repo, on top of the wire layer.  
Only the developer layer is published to package registries (PyPI, npm, Maven, etc.).

## Where the developer layer lives

Each `routeiq-sdk-{lang}` repo gains a `src/` (or `lib/`) directory alongside the
generated `proto/` output:

```
routeiq-sdk-python/
  src/
    routeiq/
      _proto/          ← generated (buf output, never edited)
        routeiq/v1/…
      sdk.py           ← hand-written developer layer  ← NEW
  pyproject.toml
  README.md
```

The published package (`pip install routeiq-sdk`) installs both. Users import from
`routeiq` (the developer layer); the proto types stay an implementation detail.

## API contract (language-agnostic)

The developer layer exposes three concepts:

| Concept | Wraps | Emitted span name |
|---|---|---|
| `Task` | `AgentEvent` + `TaskEvent` | `task:<uuid>` |
| `Step` | `AgentEvent` + `StepEvent` | `step:<uuid>` |
| `ToolCall` | `AgentEvent` + `ToolCallEvent` | `tool:<name>` |

### Initialisation

```
RouteIQ(
  agent_id:      string           # required — identifies this agent in dashboards
  otlp_endpoint: string           # default: http://localhost:4317
  tenant_id:     string           # default: "default"
  model:         string?          # populates routeiq.version.model.name
  environment:   string           # default: "production"
)
```

### Task

```
riq.task(
  intent: string,                 # routeiq.task.input_intent
  task_type: string?,             # routeiq.task.type  (e.g. "research", "coding")
) → TaskHandle
```

`TaskHandle.complete(tokens, cost_usd?, cohort?)`  
`TaskHandle.fail(category)` — `category` maps to `FailureCategory` enum

### Step

```
task.step(
  action: string?,                # routeiq.step.selected_action
  rationale: string?,             # routeiq.step.action_rationale
) → StepHandle
```

`StepHandle.complete()`  
`StepHandle.fail(category)`

### Tool call

```
step.tool(
  name: string,                   # routeiq.tool.name
  args: dict/map/object,          # hashed → routeiq.tool.arguments_hash
  permission: READ_ONLY | READ_WRITE | PRIVILEGED  # default READ_ONLY
) → ToolHandle
```

`ToolHandle.success(latency_ms?)`  
`ToolHandle.fail(error_code?, latency_ms?)`

## Language idioms

### Python — context managers

```python
from routeiq import RouteIQ

riq = RouteIQ(agent_id="my-agent", model="gpt-4o")

with riq.task(intent=user_input) as task:
    with task.step(action="tool_call") as step:
        with step.tool("search", args={"query": "Paris"}) as tool:
            result = search("Paris")
            tool.success(latency_ms=120)
    task.complete(tokens=384, cost_usd=0.002)
```

### TypeScript / JavaScript — async callbacks

```typescript
import { RouteIQ } from "routeiq-sdk";

const riq = new RouteIQ({ agentId: "my-agent", model: "gpt-4o" });

await riq.task({ intent: userInput }, async (task) => {
  await task.step({ action: "tool_call" }, async (step) => {
    const result = await step.tool("search", { query: "Paris" }, search);
    // step.tool calls search(), records success/failure automatically
  });
  task.complete({ tokens: 384, costUsd: 0.002 });
});
```

### Go — defer pattern

```go
import "github.com/SSForge/routeiq-sdk-go/routeiq"

riq := routeiq.New(routeiq.Config{AgentID: "my-agent", Model: "gpt-4o"})

task := riq.StartTask(ctx, routeiq.TaskConfig{Intent: userInput})
defer task.Complete(routeiq.TaskResult{Tokens: 384, CostUSD: 0.002})

step := task.StartStep(routeiq.StepConfig{Action: "tool_call"})
defer step.Complete()

result, err := step.RunTool("search", map[string]any{"query": "Paris"}, func() (any, error) {
    return search("Paris")
})
```

### Java — try-with-resources

```java
import io.routeiq.sdk.RouteIQ;

var riq = RouteIQ.builder().agentId("my-agent").model("gpt-4o").build();

try (var task = riq.task().intent(userInput).start()) {
    try (var step = task.step().action("tool_call").start()) {
        var result = step.tool("search")
            .arg("query", "Paris")
            .run(() -> search("Paris"));  // records success/failure
    }
    task.complete(b -> b.tokens(384).costUsd(0.002));
}
```

### Ruby — block syntax

```ruby
require "routeiq"

riq = RouteIQ::Client.new(agent_id: "my-agent", model: "gpt-4o")

riq.task(intent: user_input) do |task|
  task.step(action: "tool_call") do |step|
    result = step.tool("search", query: "Paris") { search("Paris") }
  end
  task.complete(tokens: 384, cost_usd: 0.002)
end
```

### Rust — builder + closure

```rust
use routeiq::{RouteIQ, TaskConfig, StepConfig};

let riq = RouteIQ::new("my-agent").model("gpt-4o").build();

riq.task(TaskConfig { intent: user_input, ..Default::default() }, |task| {
    task.step(StepConfig { action: "tool_call", ..Default::default() }, |step| {
        let result = step.tool("search", [("query", "Paris")], || search("Paris"));
    });
    task.complete(TaskResult { tokens: 384, ..Default::default() });
});
```

### C# — using statement

```csharp
using RouteIQ.Sdk;

var riq = new RouteIQ(new RouteIQConfig { AgentId = "my-agent", Model = "gpt-4o" });

await using var task = riq.StartTask(new TaskConfig { Intent = userInput });
await using var step = task.StartStep(new StepConfig { Action = "tool_call" });
var result = await step.RunToolAsync("search", new { query = "Paris" }, () => SearchAsync("Paris"));
await step.CompleteAsync();
await task.CompleteAsync(new TaskResult { Tokens = 384, CostUsd = 0.002m });
```

## What the developer layer does internally

Every context manager / block / defer handles exactly what the `routeiq-examples`
agent does manually today:

1. Generates `task_id`, `run_id`, `step_id` UUIDs
2. Creates an OTel span with the correct name (`task:`, `step:`, `tool:`)
3. Builds the `AgentEvent` envelope and calls `emit_agent_event(span, envelope)`
4. On exit (success or exception), calls the matching `emit_*` function and sets
   `completion_status` correctly — `SUCCESS` on clean exit, `FAILURE` on exception
5. Hashes tool arguments for `routeiq.tool.arguments_hash`
6. Records `latency_ms` automatically from wall clock if not provided

The OTel provider setup (exporter, batch processor, resource) happens once in the
`RouteIQ()` constructor and is never visible to the caller.

## What stays generated

The wire layer — all proto types, all enum values, all attribute key strings — stays
100% generated from `routeiq-schema`. The developer layer imports from the wire layer
and uses those types internally. This means:

- A new field in `StepEvent` (e.g. `retry_count`) can be exposed in the developer
  layer by adding one parameter to `step()` — the wire format update is free
- The attribute key `"routeiq.step.retry_count"` is always correct because it comes
  from the generated `routeiq_otel.py` bridge, not from a hand-written string
- All seven languages stay in sync on attribute names because they all import from
  the generated wire layer

## Why we didn't start here

The proto schema and generated SDKs needed to exist first to lock in the wire format.
Building the developer layer on top of an unstable schema would mean rewriting it
every time a field changed. Now that the schema is stable (telemetry, metrics,
insights, control, admin namespaces all defined), the developer layer can be built
with confidence that the underlying types won't move.

The `routeiq-examples` agent is effectively a prototype of the Python developer layer
written inline — it proves the design works end-to-end before we formalise it.

## Implementation order

1. **Python** — highest demand, already prototyped in `routeiq-examples`. Publish to
   PyPI as `routeiq-sdk`.
2. **TypeScript** — second most common for agent frameworks (LangChain.js, Vercel AI).
3. **Go** — common in infrastructure and tooling agents.
4. **Java, Ruby, Rust, C#** — follow the same pattern once Python is validated.

Each implementation is ~300–400 lines, lives in `src/` alongside the generated
`proto/` output in the respective `routeiq-sdk-{lang}` repo, and is tested with the
same questions used in `routeiq-examples/scripts/batch_run.py`.
